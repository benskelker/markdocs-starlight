Manually onboard Google Cloud

For use cases where you don't want to onboard your Google Cloud workspaces from the interface, this topic describes how to manually onboard a Google Cloud organization.

Before you begin

Make sure you have the following:

  • A Google Cloud organization with at least one project.

  • Google Cloud admin user that has been assigned the following permissions: 

    • roles/resourcemanager.organizationAdmin

    • roles/owner

  • The Compute Engine API enabled in .

  • We assume you are familiar with navigating your organization and projects in the Google Cloud console.

Log in to and

In the same browser, in separate tabs:

  1. Log in to your tenant.

  2. Log in to your GCP organization in the Google Cloud console.

    Keep both your tenant and the console open and stay logged in throughout the onboarding process.

Enable the Cloud Identity APIs

In this step you enable the Cloud Identity APIs in the console.

  1. In the console, navigate to API & Services > Library.

  2. In the Search for APIs & Services text box, search for Cloud Identity.

  3. From the list of APIs that appears, click the entry that matches Cloud Identity, and click ENABLE.

  4. When the API is enabled, the page reloads and shows Status as Enabled.

Add additional roles to the Google APIs Service Agent (Service account)

In this step, you add more roles to the Google APIs Service Agent.

  1. In the console, select the project where you want to deploy resources, and navigate to IAM & Admin > IAM.

  2. On the right side, select Include Google-provided role grants.

    This displays your Google-managed service accounts.

  3. In the Filter search bar (where it says Enter property name or value), search for and select the service account that ends with @cloudservices.gserviceaccount.com.

  4. Click the Edit Principal button (pencil icon) and in the page that opens:

    1. Click + ADD ANOTHER ROLE.

    2. In the Select a role drop-down, search for and select the following roles:

      • Security Admin

      • Role Administrator

    3. Save your changes.

Create the service accounts

In this step, you create service accounts for and in .

  1. In the console, select the project where you are creating all the resources and navigate to IAM & Admin > Service Accounts.

  2. One at a time, create a service account for and :

    Click + CREATE SERVICE ACCOUNTS, and provide the following details:

    Service

    Service account name

    Service account ID

    cem-service-account

    cyb-cem

    sca-service-account

    cyb-sca

  3. Click DONE.

  4. From the list of service accounts, locate each of the service accounts you just created.

  5. Create a key for each service account as follows:

    1. Select the service account.

    2. In the KEYS tab, click ADD KEY, then Create new key.

    3. For the file type, select JSON, and click CREATE.

      The generated key is automatically downloaded to your local file system. If it doesn't download, manually download and save the key to your local file system—you'll need this key later in this procedure.

  6. Navigate back to the Service Accounts page, and note the email address associated with each of the service accounts—you'll need these email addresses in the next step.

Bind the roles to their respective service accounts

In this step, you bind the roles that you created earlier to their respective service accounts by creating a new principal per service account, and assigning the relevant role to the respective principal.

  1. In the console navigate to IAM & Admin > IAM.

  2. Click Select organization, and select the organization (top level) - don't select a project.

  3. One at a time, for each service account, click +GRANT ACCESS at the top and enter the following information:

    Field

    Value

    New principals

    The email address of the service account (as noted in the previous step)

    Select a role

    • For cem-service-account: CustomCEMRole

    • For sca-service-account: CustomSCARole

  4. Save your changes. The new principal is listed with the new role in the organization's IAM page.

Generate a finalValue value for each service account

In this step you encode the each service account's key to generate a finalValue, which you'll need later in this procedure.

  1. From the console, open the Google Cloud Shell.

    • If you are prompted to authorize the Cloud Shell, authorize it.

    • Wait until the shell is fully initialized.

  2. One at a time, for each service account:

    1. In the Shell, upload the JSON file that you created when you created the service account in Create the service accounts. (For details on uploading files to the cloud shell, see the Google Cloud docs.)

    2. Run the following command which generates a Base64-encoded value of the key from the uploaded JSON file and saves it in a .txt file:

      base64 -w 0 <filename> > <base64-encoded destination file>

      For example:

      • To encode the service account key, run:

        base64 -w 0 downloaded-cv-sa-key.json > base64encoded-cv.txt

      • To encode the service account key, run:

        base64 -w 0 downloaded-sca-sa-key.json > base64encoded-sca.txt

    3. Download the saved .txt files to your local file system. You'll need them in the next step.

Onboard the Google Cloud organization in Cloud Visibility

Now that the service accounts and roles for and are configured in the console, you are ready to onboard the organization to .

  1. In , navigate to .

  2. Go to Settings > Platform Management, select the GCP tile, and click Add workspace.

  3. Select Onboard new organization and click Add. The New GCP organization page opens.

  4. Under Organization details, provide your organization's ID and the ID of the project where you created all the resources in the previous steps. Get these details from the Google Cloud console as follows:

    1. In the Google Cloud console, select the organization, copy the ID from the ID column, and paste it into the Organization ID field in the New GCP organization page in .

    2. In the Google Cloud console, select the project, copy the ID from the ID column, and paste it into the Project ID field in the New GCP organization page in .

  5. Under Configuration:

    1. Skip the first two steps.

    2. In the third step, provide the and 'finalValue values. To obtain these values, open each of the base64encoded-<xx> .txt files that you downloaded in Generate a finalValue value for each service account above. Populate each finalValue field with the Base64-encoded key from the respective .txt file.

Discover the workspace

At the bottom of the New GCP organization page, click Start Discovery.

starts to discover the organization. Depending on the size of your organization, this might take some time.

Refresh the Settings > Platform management > GCP page to check that organization onboarded as expected.

Clean up sensitive files

This is ensures that all sensitive data used in this procedure is deleted from your machine. We strongly recommend that you complete this step.

In this step, you delete the raw and encoded key files that you downloaded for the onboarding procedure.

In shell, run the following command for each sensitive file:

rm <filename>

where <filename> matches the key file, for example:

rm base64encoded-cv.txt
