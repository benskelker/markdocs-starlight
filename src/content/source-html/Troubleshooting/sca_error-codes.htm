Error messages

This topic describes the errors (listed according to error code) that are displayed in the user interface error messages and the service, so that admins can resolve issues that users encounter in .

Cloud access

These error messages appear when the end user has trouble connecting to their cloud console.

CA1001

policies only apply to users that exist in both your user directory and the cloud provider.

Verify that you added this user to the cloud provider.

CA1002

couldn't connect the user to the cloud console because it's not authorized to grant access

The onboarding for this AWS IAM Identity Center workspace wasn't completed correctly, and trust wasn't established.

  1. Sign in to the AWS management console.

  2. Go to IAM > Roles and locate the following role: CyberArkRoleSCA_{UUID}

  3. Verify that the trust information that references the role is correct.

After you apply the required information, the user will be able to connect to the cloud console.

CA1005

couldn't connect the user to the cloud console because of a problem with the user configuration

Make sure that your provisioned identity user matches the mapped account in the Google Cloud web app. For details, see Google Cloud Platform provisioning.

CC1001

AWS IAM

You attempted to onboard a workspace to , and you got an error that the workspace already exists in .

This account must have been onboarded previously to CyberArk, either in the current tenant or in a different one. So a stack already exists for this in AWS.

To onboard the account again, you must first delete the AWS stack and other CyberArk resources that were created by the CloudFormation Template (CFT) during the initial onboarding.

CC1022

 All

On-demand scan failed:

You attempted to run a scan for roles and resources but the scan failed.

Configuration issues or missing account data can cause the cloud scan to fail because can't retrieve the workspace. Make sure the credentials used to access the workspace have the correct permissions.

CC1025

All

On-demand scan only partially succeeded:

You attempted to run a scan for roles, but roles were retrieved from only some of the workspaces, while no roles were retrieved from others.

Configuration or connectivity issues can cause a partial scan - the scan stops before it has completed.

Make sure any infrastructure or account configuration changes have been synced to .

CC1026

Azure

On-demand scan for roles only partially succeeded:

You attempted to run scan for roles in the full Entra ID directory, but the resource groups and resources weren't scanned.

Access to some directory objects might be restricted due to configuration or connectivity issues.

Make sure all infrastructure or object configuration changes have been synced to .

CC1027

All

On-demand scan for roles and resources failed:

You attempted to run a scan - no roles were retrieved from any of the scanned workspaces.

Configuration issues or missing account data can cause the scan to fail because can't retrieve any roles from any of the workspaces.

Make sure the credentials being used for accessing the workspaces have the correct permissions.

policy errors

These error messages appear in a tooltip when a user hovers over an error status icon in the policy table.

Policy error codes

Error code

Environment

Scenario

Details

P1003

All

Policy not found when attempting to:

  • Get a policy's status

  • Get a policy's details

  • Delete a policy

Make sure you have provided the correct policy ID. You can get the policy ID from the audit activity event from when the policy was created.

P1021

couldn't apply the policy to the cloud console

Verify that the principal reference (for example, ARN) you used for the cloud console is correct and try again.

P1023

couldn't update the web app

In , verify the following:

  1. You created a web app that suits your use case (AWS IAM, AWS IAM Identity Center, Google Cloud, or Azure).

  2. In the Application Settings tab of the web app, you provided the following:

    • AWS IAM - the IAM account number

    • AWS IAM Identity Center - the management account number

    • Google Cloud - the domain (i.e. gcpdomain.com)

    • Azure - a valid token

  3. In the Secure Cloud Access tab, the Enable Secure Cloud Access option is checked.

P1025

All cloud providers

was unable to update the cloud configuration.

couldn't apply the policy to the cloud console. This may be due to a network timeout, connectivity issues, or a permissions issue.

To resolve permissions issues, verify the following:

  • AWS - the policies for CyberArkRoleSCA_{UUID} haven't been modified, and that the role has a trust relationship with the account.

  • Azure and Google Cloud - the service user has not expired and is configured properly.

P1026

The policy reached the maximum number of role bindings for your allow policy.

  • For group-based assignment - enforces a limit of role bindings to groups. Verify that the number of roles in your SCA policy doesn't exceed the maximum role binding limit defined in .

    For more information about Google Cloud limits, see the docs.

  • For direct user assignments - not relevant.

To resolve the issue, delete old and unused polices to free up space.

P1028

AWS

couldn't add this identity to the policy because there's no custom role associated with the relevant web app.

Create a custom role for the relevant web app in the format AWS_SCA_<management account ID>, then add the identity again.

P1032

AWS

couldn't update the cloud configuration for a standalone AWS account / AWS organization.

couldn't validate the policy to the cloud console.

One of the cloud roles in the policy is missing the required trust relationship with the correct IdP, which is necessary for to process JIT Connect.

P1034

All cloud providers

couldn't complete the validation process because of an error related to cloud roles.

In the cloud provider, verify the cloud roles that are assigned to the policy. Update the policy accordingly and validate it again.

P1035

All cloud providers

couldn't complete the validation process because it couldn't find some of the identities.

In your IdP, verify the identities that are relevant for the policy. Update the policy accordingly and validate it again.

P1036

AWS

couldn't validate the policy because a required AWS permission is blocked at the organizational level by an AWS service control policy.

Unblock the permission and try to validate the policy again. For a list of required permissions, see the AWS required permissions.

P1038

couldn't create the policy

The Google Cloud policy creation failed because permissions couldn't be propagated to the workspace.
